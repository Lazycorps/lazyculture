# Bibliothèque, Librairie et Événements

Ces 3 salles spéciales (+ le type `SHOP`/`REST`/`EVENT`) partagent un patron commun : `chooseNode` les passe en `ACTIVE` sans tirer de question, une méthode de résolution dédiée dans `BrainrunService` les fait passer en `CLEARED`, puis `acknowledgeRoom` avance la run (sauf `SHOP` qui avance directement via `leaveShop`).

## ⚠️ Nommage : ne pas confondre avec les commentaires du code

Les libellés joueur sont définis dans `app/composables/useBrainrunRoomTypeDisplay.ts` :

- `REST` → **"Bibliothèque"**
- `SHOP` → **"Librairie"** (mais les commentaires de `BrainrunService.ts` l'appellent presque partout "Boutique" — incohérence de nommage interne/UI à connaître, pas une distinction fonctionnelle : c'est la même salle)

## Bibliothèque (`REST`)

Résolue par `BrainrunService.resolveRest(runId, choice, theme?)`. Contrairement aux autres salles instantanées, elle **exige toujours un choix explicite** (reste `ACTIVE` tant que non résolue) :

- `"HEAL"` — régénère 1 PV (`instantRoomHealthDelta("REST")`), plafonné à `maxHealthPoint`.
- `"BAN_THEME"` — bannit un thème pour le reste de la run (`run.bannedThemes`), en s'assurant qu'il fait partie de `availableThemesToBan` (calculé par `computeAvailableThemesToBan` : union des thèmes de tous les ennemis/boss, moins `culture_generale` — **jamais bannissable** car systématique dans les boss — et moins les thèmes de `computeUnsafeToBanThemes` — un thème partagé par **tous** les boss d'un même acte, ex. `anime-manga` pour l'Acte 2, viderait entièrement le pool de boss de cet acte si banni). Une fois banni, ce thème est exclu du tirage d'ennemis/boss (même filtre `notBanned` que la relique Purge Thématique — voir `items.md`) **et** directement des questions tirées (`bannedThemes` passé en `excludeThemes` à `QuestionService.getRandomIdsByDifficulty`) — indispensable car une question peut porter plusieurs thèmes à la fois, donc filtrer uniquement sur les thèmes de l'ennemi/boss ne suffit pas à garantir qu'un thème banni n'apparaisse jamais.

## Librairie (`SHOP`)

- **Entrée** (`chooseNode`) — génère les offres via `generateShopOffers` : `BRAINRUN_SHOP_RELIC_OFFER_COUNT` (2) reliques non possédées (priorité par rareté, prix par rareté : `BRAINRUN_RELIC_SHOP_PRICE` COMMON 40 / RARE 90) + `BRAINRUN_SHOP_CONSUMABLE_OFFER_COUNT` (2) consommables (prix fixe par item, `shopPrice`). Si le pool de reliques non possédées est épuisé, l'emplacement devient une offre d'or gratuit (`BRAINRUN_GOLD_FALLBACK_OFFER_AMOUNT` = 15) plutôt qu'une relique déjà possédée.
- **Achat** (`buyShopItem(offerIndex)`) — débit d'or **atomique et conditionnel** (`updateMany` avec `gold: { gte: price }`) pour éviter qu'un double-clic achète à crédit ; c'est la seule opération Brainrun protégée ainsi avec le déblocage de talent (le reste du service est lecture-puis-écriture, risque mineur assumé). Relique **Marchandeur** (`HAGGLER`) : -20% sur tous les prix (`BRAINRUN_HAGGLER_MULTIPLIER` = 0.8). Relique **Fournisseur Fidèle** (`RESTOCK`) : l'offre achetée est remplacée par une nouvelle du même type (`generateShopReplacementOffer`) plutôt que retirée — le joueur peut acheter en boucle tant qu'il a l'or.
- **Sortie** (`leaveShop`) — marque `CLEARED` et avance directement (pas d'`acknowledgeRoom` intermédiaire, contrairement aux salles de combat/Bibliothèque). Bloqué si un choix de thème banni est en attente (Purge Thématique achetée juste avant).
- Reliques jamais vendues en Librairie : `REVIVE_TOKEN` n'a pas de `shopPrice` (uniquement gagnable en bonus post-combat/Événement) — volontaire, pour ne pas trivialiser une run mal engagée avec de l'or.

## Événements (`EVENT`)

Refonte du 2026-07-12 (façon Slay the Spire). Catalogue `BRAINRUN_EVENTS` (`shared/brainrunItems.ts`) — **24 événements, 8 par acte** (champ `act`), montée en puissance : Acte 1 petits enjeux (or/PV/communs), Acte 2 moyens (RARE, 2 charges de Bouclier, 50/50 x3), Acte 3 gros (relique, résurrection contre PV max, effets premium). `getBrainrunEventIdsByAct(act)` donne le pool d'un acte.

### Structure d'un événement

- `BrainrunEventDef` : `id`, `act`, `title`, `description`, `options: BrainrunEventOption[]` (**2 à 3 options**, plus le tuple figé à 2 d'avant — le client les affiche déjà en boucle `v-for`).
- `BrainrunEventOption` : `label`, `cost?` (**explicite, payé d'avance**), `outcomes: BrainrunEventOutcome[]` (**≥ 1**).
- `BrainrunEventOutcome` : `weight` (poids relatif), `reward?` (absent = "rien ne se passe"), `resultText` (lore affiché au passé). Le résultat est **tiré au sort masqué** parmi la table pondérée (`weightedRandomPick`) — c'est le "ratio de chance positif". Une option à 1 seul outcome est déterministe.
- **Convention anti-frustration** : un `outcome` ne porte **jamais** de coût caché. Toute perte (or/PV/PV max/relique) est explicite sur `cost`, visible avant de choisir. Les outcomes ne font qu'ajouter du positif ou rien — un tirage masqué ne peut donc **jamais** tuer ni surprendre par une perte.
- `BrainrunEventReward` étendu : `hp`, **`fullHeal`** (restaure tous les PV, prioritaire sur `hp` ; appliqué par `resolveEvent` après stabilisation du PV max), `gold`, `relic:"RANDOM"`, `consumables[]`, **`shieldCharges`** (charges de Bouclier), **`fiftyFiftyCharges`** (50/50 auto sur N prochaines questions), **`revive`** (octroie un `REVIVE_TOKEN`). `BrainrunEventCost` étendu : `hp`, `gold`, **`maxHp`** (réduction permanente de PV max), `relic:"RANDOM_OWNED"`.

### Cycle de vie

- **eventId fixé à la génération** (comme les ennemis depuis le 2026-07-09) : `seedActGraph` appelle `assignEventIdentities(eventNodes, getBrainrunEventIdsByAct(act), random)` — tirage **sans remise** dans le pool de l'acte, ce qui **garantit aucun doublon d'Événement sur une run** tant que le pool (8) couvre le nombre de nœuds EVENT (retombe sur des répétitions au-delà). `resolveNodeChoice` ne tire plus rien à l'entrée : il lit `node.eventId`, avec filet `pickFallbackEventId` si absent (nœud EVENT forcé via `debugJumpToNode`, ou run antérieure à la migration dont les nœuds PENDING n'avaient pas d'eventId).
- **Résolution** (`resolveEvent(optionIndex)`) — délègue à `resolveEventOption` (`brainrunLogic.ts`, pure/testable) : tire l'outcome masqué, calcule `hpDelta`/`goldDelta`/`maxHpDelta`, `shieldChargesGranted`/`fiftyFiftyChargesGranted`/`reviveGranted`, la relique octroyée (exclut les possédées, retombe sur de l'or si le pool est épuisé) et la relique sacrifiée (`RANDOM_OWNED`), et renvoie `outcomeIndex`+`resultText`. Le service applique : Bouclier consommé pour un coût PV (`consumeShieldCharge`), PV max abaissé par `maxHp` puis relevé par `EXTRA_HEART`, charges de Bouclier gagnées (`grantShieldCharge`, plafonnées aux PV), `fiftyFiftyCharges` incrémentées, mort éventuelle avec le même filet consommable→relique→talent qu'en combat, et les octrois (`recordRelicDiscovery`, Purge Thématique en attente, `grantConsumable` pour les consommables).
- **Coût `maxHp`** refusé (400 serveur **et** option désactivée client) si le PV max passerait sous `BRAINRUN_EVENT_MIN_MAX_HP` (2). **Résurrection (`revive`)** octroyée par **incrément direct** du `REVIVE_TOKEN` (hors plafond d'emplacements `grantConsumable`) : récompense garantie payée en PV max, elle ne doit jamais être perdue faute de place — seule entorse assumée au plafond de consommables (affichage HUD éventuellement au-delà des slots, cosmétique).
- **Affichage** — salle EVENT `CLEARED` rendue par `BrainrunEvent.vue` : le `resultText` (lore) + une liste de chips du détail réellement gagné/perdu (dérivée des champs de `eventOutcome`, dont `shieldChargesGranted`/`fiftyFiftyChargesGranted`/`reviveGranted`/`maxHpDelta`), + bouton "Continuer" → `acknowledgeRoom()`. Les boutons de choix n'affichent **que le coût** (`summarizeCost`), la récompense restant masquée. `BrainrunRoomDTO.eventOutcome` (`BrainrunEventOutcomeDTO`, persisté par le serveur) est la source — ne pas reconstruire le texte depuis l'option statique côté client.
- **50/50 automatique** (Lot B) : `BrainrunRun.fiftyFiftyCharges: Int` (champ Prisma, migration `20260712120000_brainrun_fifty_fifty_charges`). Chaque charge applique un 50/50 à la **prochaine question de combat présentée** puis se consomme, via `computeQuestionEntryReveal(questionId, effects, fiftyFiftyChargesAvailable)` (remplace l'ancien `computeAutoHintReveal`, cumule Sixième Sens + 50/50) — branchée aux 2 points d'entrée de question (entrée en combat dans `resolveNodeChoice`, question suivante dans `submitAnswer`), qui décrémentent `fiftyFiftyCharges` quand `fiftyFiftyConsumed`. **Persistent entre les combats** (contrairement au Bouclier). Exposé au HUD (`app/pages/brainrun/index.vue`, icône ciseaux). Edges connus assumés : REDRAW efface le 50/50 auto de la question redistribuée (charge déjà consommée) ; combinaison avec le malus d'intro d'Alain non spécialement gérée (jank mineur).
- Relique **Aimant à Événements** (`EVENT_MAGNET`) : convertit rétroactivement les nœuds `STANDARD`/`ELITE` pas encore atteints en `EVENT` (`convertUpcomingNodesToEvents`) — assigne désormais aussi un `eventId` par nœud converti, en **excluant les événements déjà présents** sur la carte de l'acte pour préserver la garantie anti-doublon.

## Si tu ajoutes une salle spéciale ou modifies celles-ci

- Un nouvel événement : ajouter une entrée à `BRAINRUN_EVENTS` avec son `act` et 2-3 `BrainrunEventOption`, chaque option portant une table `outcomes` pondérée. La logique (`resolveEventOption`) est déjà générique sur tous les coûts/récompenses ci-dessus (hp/gold/maxHp/relic/consumables/shield/50-50/revive) — pas de nouveau code sauf coût/récompense vraiment inédit. Garder les pertes sur `cost` (jamais dans un outcome). Vérifier que le nombre d'événements par acte reste ≥ au nombre de nœuds EVENT probables (sinon des doublons réapparaissent, cf. fallback d'`assignEventIdentities`).
- Un nouveau type de salle instantanée : suivre le patron Bibliothèque/Librairie — statut `ACTIVE` tant que non résolu, méthode de résolution dédiée dans `BrainrunService`, décider si elle avance directement (comme `leaveShop`) ou via `acknowledgeRoom` (comme `resolveRest`/`resolveEvent`), et l'ajouter à `BRAINRUN_INSTANT_ROOM_TYPES` + aux quotas de `map.md`.
